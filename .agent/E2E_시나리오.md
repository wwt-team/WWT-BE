# E2E 시나리오

## 목적

이 문서는 도메인별 API를 따로 검증하는 수준을 넘어서, 사용자가 실제로 서비스를 이용하는 흐름을 처음부터 끝까지 통으로 검증하기 위한 시나리오를 정리한다.

기준은 아래와 같다.

- 사용자 행동 순서대로 본다.
- 각 단계의 성공 결과가 다음 단계의 입력이 된다.
- 단순 상태코드보다 실제 상태 전이와 연결성을 본다.

---

## 1. Auth E2E

### 시나리오

1. 이메일 인증 코드 요청
2. 이메일 인증 코드 검증
3. 회원가입
4. 로그인
5. access token으로 보호 API 접근 확인
6. refresh
7. logout
8. logout 이후 기존 refresh token 차단 확인

### 확인 포인트

- 회원가입 흐름이 끝까지 이어지는가
- 로그인 후 보호 API가 실제로 열리는가
- refresh 후 세션이 유지되는가
- logout 후에는 이전 세션이 더 이상 유효하지 않은가

---

## 2. Users E2E

### 시나리오

1. 로그인
2. `GET /api/users/me`
3. `GET /api/users/me/products`

### 확인 포인트

- 로그인 사용자 정보가 실제 응답으로 내려오는가
- 내 상품 목록이 내 계정 컨텍스트와 맞게 내려오는가

---

## 3. Products E2E

### 판매자 흐름

1. 로그인
2. 상품 등록
3. 상품 상세 조회
4. 상품 수정
5. 상품 상태 변경
6. 상품 삭제

### 구매자 흐름

1. 로그인
2. 상품 목록 조회
3. 상품 검색
4. 상품 상세 조회

### 확인 포인트

- 등록한 상품이 상세/목록에서 같은 정보로 보이는가
- 수정 결과가 상세에 반영되는가
- 상태 변경 결과가 이후 조회에 반영되는가
- 동일 판매자 + 같은 제목 + 같은 설명의 중복 등록이 차단되는가

---

## 4. Trade Requests E2E

### 시나리오

1. 구매자 로그인
2. 상품 상세 조회
3. 거래 요청 생성
4. 내 거래 요청 목록 조회
5. 판매자 로그인
6. 상품별 거래 요청 목록 조회
7. 거래 요청 수락 또는 거절
8. 수락 후 상품 상태 확인

### 확인 포인트

- 요청 생성부터 판매자 처리까지 한 흐름으로 이어지는가
- 수락 시 상품 상태가 `RESERVED`로 바뀌는가
- 이미 처리된 요청은 다시 처리되지 않는가

---

## 5. Chats E2E

### REST 흐름

1. 로그인
2. 채팅방 생성
3. 채팅방 목록 조회
4. 메시지 목록 조회
5. 메시지 생성
6. 메시지 수정
7. 메시지 삭제

### WebSocket 흐름

1. 소켓 연결
2. 채팅방 입장
3. 메시지 생성 이벤트 수신
4. 메시지 수정 이벤트 수신
5. 메시지 삭제 이벤트 수신

### 확인 포인트

- 채팅방 생성 후 목록에서 바로 보이는가
- 메시지 생성/수정/삭제 결과가 목록에 일관되게 반영되는가
- 실시간 이벤트가 참여자에게만 전달되는가

---

## 6. Deploy QA와의 관계

e2e는 테스트 환경에서 통짜 흐름을 검증하는 것이고, deploy QA는 실제 EC2/Nginx/SMTP/WebSocket 환경에서 같은 흐름이 운영 주소에서 동작하는지 확인하는 것이다.
