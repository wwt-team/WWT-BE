# 테스트 케이스 분류표

## 분류 기준

### Fast

- 사용자가 실제로 자주 밟는 핵심 흐름
- 한두 단계 안에서 빠르게 검증 가능
- 외부 의존성이 거의 없음
- PR마다 돌려도 부담이 작음

### Complex

- 사용자는 평범하게 사용한다고 생각하지만, 시스템 입장에서는 꼬이기 쉬운 규칙
- 여러 단계가 연결되어 상태가 함께 바뀌는 흐름
- 중복 시도, 재시도, 재사용, 이미 끝난 상태 재처리 같은 경계 상황
- WebSocket, token rotation처럼 규칙이 눈에 잘 보이지 않는 영역

### Deploy QA

- 실제 배포 환경에서 사람이 직접 확인해야 하는 항목
- EC2, Nginx, PM2, CORS, SMTP, WebSocket, HTTPS 같은 운영 의존성 포함

## 1. Auth

### Fast 시나리오

1. `signup/email-code`
2. `signup/email-code/verify`
3. `signup`
4. `login`
5. `accessToken`으로 보호 API 접근 확인
6. `refresh`
7. `logout`

### Fast 검증 포인트

- 이메일 형식 오류
- 이미 가입된 이메일
- 인증 코드 불일치
- 인증 코드 만료
- 비밀번호 형식 오류
- 닉네임 누락
- 이메일 인증 토큰 누락
- 이메일 인증 토큰 만료/재사용
- 로그인 불일치
- access token으로 보호 API 접근 성공
- 인증 없이 보호 API 접근 실패

### Complex

#### 사용자 상황

- 사용자는 로그인 상태를 유지하려고 refresh를 여러 번 시도할 수 있다.
- 로그아웃 후에도 앱이 예전 refresh token으로 다시 연장 요청을 보낼 수 있다.
- 오래된 토큰이 남아 있는 상태에서 재사용이 발생할 수 있다.

#### 검증해야 할 규칙

- refresh token이 만료되면 재로그인을 요구해야 한다.
- rotation 이후 이전 refresh token은 다시 사용할 수 없어야 한다.
- revoked token 재사용은 일반 만료와 다르게 감지되어야 한다.
- logout 이후 기존 refresh token은 반드시 막혀야 한다.

### Deploy QA

- 실제 이메일 수신 확인
- 배포 환경에서 `signup -> login -> refresh -> logout` 전체 흐름 확인

## 2. Users

### Fast 시나리오

1. `login`
2. `GET /api/users/me`
3. `GET /api/users/me/products`

### Fast 검증 포인트

- 인증 없음
- 내 정보 정상 조회
- 내 상품 목록 정상 조회

### Complex

#### 사용자 상황

- 내 상품이 많아지면 페이지를 넘기며 확인하게 된다.
- 목록이 길어져도 중복 없이 정확한 순서로 보여야 한다.

#### 검증해야 할 규칙

- pagination이 바뀌어도 아이템이 중복되거나 빠지지 않아야 한다.
- `page`, `limit`, `total`이 실제 데이터와 일치해야 한다.

### Deploy QA

- 실제 배포 환경에서 내 정보/내 상품 목록 흐름 확인

## 3. Products

### Fast 시나리오

#### 판매자 흐름

1. `login`
2. 상품 등록
3. 상품 상세 조회
4. 상품 수정
5. 상품 상태 변경
6. 상품 삭제

#### 구매자 흐름

1. `login`
2. 상품 목록 조회
3. 상품 검색
4. 상품 상세 조회

### Fast 검증 포인트

- 목록 page/limit 검증
- 검색 status/min/max/range 검증
- 상세 not found
- 등록 시 인증 없음
- 제목 누락
- 가격 오류
- 이미지 URL 오류
- 수정/삭제/상태 변경 시 판매자 권한 검증
- 상품 not found

### Complex

#### 사용자 상황

- 판매자가 상품 상태를 바꾸면 목록과 상세에서도 같은 상태를 기대한다.
- 구매자는 검색 조건을 여러 개 섞어서 사용한다.
- 판매자가 상태를 바꿨는데 검색 결과가 이전 상태로 남아 있으면 혼란이 생긴다.
- 동일 판매자가 같은 제목과 같은 설명의 상품을 실수로 두 번 등록하려고 할 수 있다.

#### 검증해야 할 규칙

- 상품 상태 변경 후 목록과 상세 응답이 일관되어야 한다.
- 검색 조건을 여러 개 적용해도 결과가 정확해야 한다.
- `ON_SALE`, `RESERVED`, `SOLD` 상태가 이후 조회와 거래 흐름에 맞게 반영되어야 한다.
- 동일 판매자가 같은 제목과 같은 설명의 상품을 다시 등록하려고 하면 차단되어야 한다.

### Deploy QA

- 배포 URL에서 목록/검색/상세/등록/수정/삭제/상태 변경 전체 흐름 확인

## 4. Trade Requests

### Fast 시나리오

#### 구매자 흐름

1. `login`
2. 상품 상세 조회
3. 거래 요청 생성
4. 내 거래 요청 목록 조회

#### 판매자 흐름

1. `login`
2. 상품별 거래 요청 목록 조회
3. 거래 요청 수락 또는 거절

### Fast 검증 포인트

- 인증 없음
- 상품 없음
- 자기 상품 요청 차단
- 판매 완료 상품 차단
- 중복 대기 요청 차단
- 판매자 아님 차단
- 요청 없음
- `PENDING`이 아닌 요청 차단

### Complex

#### 사용자 상황

- 구매자는 같은 상품에 버튼을 여러 번 누를 수 있다.
- 판매자는 이미 처리한 요청을 다시 클릭할 수 있다.
- 판매자가 요청을 수락하면 상품이 자동으로 예약 상태가 되길 기대한다.

#### 검증해야 할 규칙

- 같은 상품에 `PENDING` 요청이 중복 생성되지 않아야 한다.
- 거래 요청 수락 시 상품 상태가 `RESERVED`로 함께 바뀌어야 한다.
- 이미 수락/거절된 요청은 다시 처리할 수 없어야 한다.
- 요청 상태와 상품 상태가 서로 모순되지 않아야 한다.

### Deploy QA

- 구매자와 판매자 계정으로 실제 거래 요청 생성/수락/거절 흐름 확인

## 5. Chats

## 5-1. Chats REST

### Fast 시나리오

1. `login`
2. 채팅방 생성
3. 채팅방 목록 조회
4. 메시지 목록 조회
5. 메시지 전송
6. 메시지 수정
7. 메시지 삭제

### Fast 검증 포인트

- `productId` 누락
- `receiverId` 누락
- `receiverId` 형식 오류
- 인증 없음
- 상품 없음
- 자기 자신과 채팅 차단
- 참여자 아님 차단
- 빈 메시지 차단
- 삭제된 메시지 수정 차단
- 메시지 없음

### Complex

#### 사용자 상황

- 같은 상품, 같은 상대와 채팅을 여러 번 열려고 시도할 수 있다.
- 메시지를 삭제한 뒤 다시 수정 버튼을 누를 수 있다.
- 채팅방 목록에서 마지막 메시지가 즉시 반영되길 기대한다.

#### 검증해야 할 규칙

- 동일 상품/상대 기준으로 채팅방이 불필요하게 중복 생성되지 않아야 한다.
- 삭제된 메시지는 다시 수정할 수 없어야 한다.
- 마지막 메시지와 채팅방 상태가 메시지 수정/삭제 이후에도 일관되어야 한다.

### Deploy QA

- 배포 환경에서 채팅방 생성부터 메시지 수정/삭제까지 실제 흐름 확인

## 5-2. Chats WebSocket

### Fast

- 없음

### Complex

#### 사용자 상황

- 사용자는 참여자가 아닌 방에 들어가려고 할 수 있다.
- 메시지를 보내면 상대방 화면에 바로 보이길 기대한다.
- 토큰이 잘못되었을 때도 연결을 시도할 수 있다.

#### 검증해야 할 규칙

- 잘못된 토큰 연결은 거부되어야 한다.
- 참여자만 `join_room` 할 수 있어야 한다.
- `send/update/delete` 이벤트는 참여자에게만 전달되어야 한다.
- 참여자가 아닌 사용자는 이벤트를 받거나 보낼 수 없어야 한다.

### Deploy QA

- 브라우저 2개 이상으로 실제 실시간 메시지 반영 확인
- 외부 네트워크 환경에서 연결 유지와 이벤트 수신 확인

## 6. Deploy QA

### Fast

- 없음

### Complex

#### 사용자 상황

- 사용자는 서버가 재부팅돼도 서비스가 계속 살아 있기를 기대한다.
- 브라우저에서는 API, 소켓, 메일 발송이 모두 실제 운영 주소에서 동작해야 한다.

#### 검증해야 할 규칙

- PM2 프로세스가 죽지 않고 유지되어야 한다.
- 서버 재부팅 후 PM2가 자동 시작되어야 한다.
- Nginx reverse proxy가 지속적으로 정상 동작해야 한다.

### Deploy QA

1. `http://<server>/api/products` 외부 호출
2. Auth 전체 흐름 확인
3. Users 흐름 확인
4. Products 흐름 확인
5. Trade Requests 흐름 확인
6. Chats REST 흐름 확인
7. WebSocket 실시간 흐름 확인
8. CORS 허용 확인
9. SMTP 실제 발송 확인
10. HTTPS 적용 후 인증서/리다이렉트 확인
