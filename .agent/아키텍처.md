# 중고거래 API 아키텍처

## 1. 기본 방향

- Framework: NestJS
- Package Manager: pnpm
- API Style: REST API
- Database: PostgreSQL
- ORM: TypeORM
- Auth: JWT Bearer 인증
- Token: accessToken + refreshToken, RFR 방식
- 구조: NestJS의 기본 `module-controller-service` 구조를 따른다.

이 프로젝트는 기능별 모듈을 `src/modules` 아래에 나누고, 공통 설정과 공통 유틸은 `src/common`, `src/config`, `src/database`에 둔다.

세부 구현에서는 도메인 안에서도 기능별 폴더를 나눈다. 예를 들어 Auth 도메인은 `features/login`, `features/signup`, `features/refresh`처럼 분리한다. 자세한 규칙은 `.agent/개발규칙.md`를 따른다.

## 3. 계층 역할

### Controller

HTTP 요청과 응답을 담당한다.

- 라우팅 정의
- DTO 검증 요청 수신
- 인증 guard 적용
- service 호출
- HTTP status code 반환

Controller에는 비즈니스 로직을 넣지 않는다.

예:

```ts
@Post('login')
login(@Body() dto: LoginDto) {
  return this.authService.login(dto);
}
```

### Service

비즈니스 로직을 담당한다.

- 로그인 검증
- 비밀번호 해시 비교
- 토큰 발급
- 상품 권한 검사
- 거래 요청 상태 변경
- 채팅방 참여자 검증
- repository 호출

Service는 controller에 의존하지 않는다.

### Entity

TypeORM 테이블 모델이다.

- DB 컬럼 정의
- relation 정의
- enum 컬럼 정의
- index, unique 제약 정의

### DTO

요청 body/query/param 검증용 객체다.

- `class-validator`
- `class-transformer`
- 요청값 타입 변환
- 필수값 검증

### Common

여러 모듈에서 같이 쓰는 코드다.

- 공통 에러 코드
- 공통 예외 필터
- JWT Guard
- 현재 사용자 decorator
- password hash util

## 4. 모듈 구성

## AuthModule

담당 API:

- `POST /api/auth/signup/email-code`
- `POST /api/auth/signup/email-code/verify`
- `POST /api/auth/signup`
- `POST /api/auth/login`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`

주요 책임:

- 이메일 인증 코드 발송 및 검증
- 회원가입
- 로그인
- accessToken 발급
- refreshToken 발급
- refreshToken 해시 저장
- RFR 방식 refreshToken 교체
- 로그아웃 시 refreshToken 폐기

의존:

- `UsersService`
- `JwtService`
- `RefreshToken` repository
- `EmailVerification` repository

로그인 흐름:

1. `AuthController.login()`이 `LoginDto`를 받는다.
2. `AuthService.login()`이 email로 user를 조회한다.
3. user가 없거나 비밀번호가 틀리면 `INVALID_CREDENTIALS`를 반환한다.
4. 비밀번호가 맞으면 accessToken, refreshToken을 발급한다.
5. refreshToken 원문은 응답으로만 내려주고 DB에는 hash만 저장한다.
6. user 정보와 토큰을 반환한다.

## 5. 인증 구조

### Access Token

- API 인증에 사용한다.
- `Authorization: Bearer {accessToken}` 헤더로 전달한다.
- payload에는 최소한 `sub`, `email`을 넣는다.

예:

```json
{
  "sub": 1,
  "email": "user@example.com"
}
```

### Refresh Token

- accessToken 재발급에만 사용한다.
- DB에는 원문을 저장하지 않고 hash만 저장한다.
- refreshToken마다 token family를 가진다.
- refresh 성공 시 기존 refreshToken을 폐기하고 새 refreshToken을 저장한다.

### Guard

인증이 필요한 API는 `JwtAuthGuard`를 사용한다.

```ts
@UseGuards(JwtAuthGuard)
@Get('me')
getMe(@CurrentUser() user: RequestUser) {
  return this.usersService.getMe(user.id);
}
```

### CurrentUser Decorator

JWT payload에서 현재 로그인 사용자를 꺼내기 위해 사용한다.

```ts
export type RequestUser = {
  id: number;
  email: string;
};
```

## 6. 공통 에러 응답

모든 에러는 API 명세의 형식으로 내려간다.

```json
{
  "statusCode": 400,
  "code": "INVALID_LOGIN_EMAIL",
  "message": "이메일 형식이 올바르지 않습니다.",
  "timestamp": "2026-04-15T12:00:00.000Z",
  "path": "/api/auth/login"
}
```

구현 방식:

- 기능별 service에서 `HttpException` 또는 커스텀 예외를 던진다.
- `HttpExceptionFilter`가 응답 형식을 통일한다.
- `code`는 `common/constants/error-codes.ts`에서 관리한다.

## 8. pnpm 명령어

프로젝트 생성:

```powershell
pnpm dlx @nestjs/cli new .
```

패키지 설치:

```powershell
pnpm add @nestjs/config @nestjs/typeorm typeorm pg
pnpm add @nestjs/jwt @nestjs/passport passport passport-jwt bcrypt
pnpm add class-validator class-transformer
pnpm add -D @types/passport-jwt @types/bcrypt
```

개발 서버:

```powershell
pnpm start:dev
```

빌드:

```powershell
pnpm build
```

테스트:

```powershell
pnpm test
```
