# 개발 규칙

## 1. 기본 원칙

이 프로젝트는 NestJS의 기본 구조를 따른다.

- 기능은 도메인별 module로 나눈다.
- 도메인 안에서는 세부 기능별 폴더를 다시 나눈다.
- controller는 HTTP 입출력만 담당한다.
- service는 비즈니스 로직을 담당한다.
- DTO는 기능별 폴더 안에 둔다.
- entity는 도메인 공통 데이터 모델이므로 도메인 루트의 `entities` 폴더에 둔다.
- 공통으로 쓰는 guard, decorator, filter, util은 `src/common`에 둔다.

핵심 기준은 "파일 위치만 봐도 어떤 API 기능인지 알 수 있게 만든다"이다.

## 2. 도메인별 구조 규칙

도메인은 `src/modules/{domain}` 아래에 만든다.

예:

```txt
src/modules/
  auth/
  users/
  products/
  trade-requests/
  chats/
```

각 도메인은 기본적으로 아래 구조를 가진다.

```txt
{domain}/
  {domain}.module.ts
  {domain}.controller.ts
  {domain}.service.ts
  entities/
  dto/
```

단, 도메인 안의 기능이 2개 이상이면 기능별 폴더를 만든다.

```txt
{domain}/
  {domain}.module.ts
  {domain}.controller.ts
  {domain}.service.ts
  entities/
  features/
    {feature}/
      dto/
      {feature}.service.ts
```

## 3. Auth 도메인 구조

Auth는 기능이 명확히 나뉘므로 `features` 아래에 기능별로 분리한다.

```txt
src/modules/auth/
  auth.module.ts
  auth.controller.ts
  auth.service.ts

  entities/
    refresh-token.entity.ts
    email-verification.entity.ts

  strategies/
    jwt.strategy.ts

  features/
    login/
      dto/
        login.dto.ts
      login.service.ts

    signup/
      dto/
        signup.dto.ts
      signup.service.ts

    email-verification/
      dto/
        request-email-code.dto.ts
        verify-email-code.dto.ts
      email-verification.service.ts

    refresh/
      dto/
        refresh-token.dto.ts
      refresh-token.service.ts

    logout/
      dto/
        logout.dto.ts
      logout.service.ts
```

### AuthController 역할

AuthController는 auth 관련 endpoint를 한 곳에서 받는다.

```txt
POST /api/auth/login
POST /api/auth/signup
POST /api/auth/signup/email-code
POST /api/auth/signup/email-code/verify
POST /api/auth/refresh
POST /api/auth/logout
```

Controller는 각 feature service로 위임한다.

예:

```ts
@Post('login')
login(@Body() dto: LoginDto) {
  return this.loginService.login(dto);
}
```

### AuthService 역할

`auth.service.ts`는 auth 전체에서 공유되는 보조 로직만 둔다.

예:

- accessToken 생성
- refreshToken 생성
- refreshToken hash 생성
- token payload 생성

로그인, 회원가입, 로그아웃 같은 기능별 로직은 각 feature service에 둔다.

## 4. Users 도메인 구조

```txt
src/modules/users/
  users.module.ts
  users.controller.ts
  users.service.ts
  entities/
    user.entity.ts
  features/
    me/
      me.service.ts
    my-products/
      dto/
        my-products-query.dto.ts
      my-products.service.ts
```

규칙:

- 사용자 생성/조회 같은 공통 로직은 `users.service.ts`에 둔다.
- `GET /api/users/me` 전용 로직은 `features/me`에 둔다.
- `GET /api/users/me/products` 전용 로직은 `features/my-products`에 둔다.

## 8. 파일명 규칙

파일명은 기능이 드러나게 작성한다.

좋은 예:

```txt
login.service.ts
signup.service.ts
create-product.service.ts
update-product-status.service.ts
accept-trade-request.service.ts
chat-messages.service.ts
```

나쁜 예:

```txt
service.ts
handler.ts
logic.ts
util.ts
```

## 9. DTO 규칙

DTO는 해당 기능 폴더 안에 둔다.

예:

```txt
auth/features/login/dto/login.dto.ts
products/features/create/dto/create-product.dto.ts
products/features/search/dto/product-search-query.dto.ts
```

DTO 이름은 요청 목적이 드러나게 작성한다.

```ts
export class LoginDto {}
export class SignupDto {}
export class CreateProductDto {}
export class ProductSearchQueryDto {}
```

## 10. Entity 규칙

Entity는 feature 폴더에 넣지 않는다.

Entity는 도메인 전체에서 공유되는 DB 모델이므로 도메인 루트의 `entities` 폴더에 둔다.

```txt
src/modules/auth/entities/refresh-token.entity.ts
src/modules/users/entities/user.entity.ts
src/modules/products/entities/product.entity.ts
```

## 11. 공통 로직 분리 기준

아래에 해당하면 도메인 루트 service로 올린다.

- 같은 도메인의 feature 2개 이상에서 사용한다.
- DB entity 조회나 권한 확인처럼 도메인 공통 의미가 있다.
- 다른 도메인에서 호출해야 한다.

예:

```txt
products.service.ts
  findProductOrThrow()
  assertSeller()
  updateProductStatus()

chats.service.ts
  findRoomOrThrow()
  assertParticipant()
```

아래에 해당하면 feature service에 둔다.

- 특정 API 하나에서만 사용한다.
- 요청/응답 형식에 강하게 묶여 있다.
- 기능 이름 없이 의미가 애매하다.

## 12. Import 규칙

도메인 간 의존은 가능한 service를 통해서만 한다.

예:

```ts
TradeRequestsService -> ProductsService
ChatsService -> ProductsService
```

다른 도메인의 repository를 직접 주입하지 않는다.

나쁜 예:

```ts
TradeRequestsService -> ProductRepository 직접 주입
```

좋은 예:

```ts
TradeRequestsService -> ProductsService.findProductOrThrow()
```

## 13. 로그인 구현 시 적용 예시

처음 로그인만 구현할 때 필요한 구조:

```txt
src/modules/auth/
  auth.module.ts
  auth.controller.ts
  auth.service.ts
  features/
    login/
      dto/
        login.dto.ts
      login.service.ts
  strategies/
    jwt.strategy.ts

src/modules/users/
  users.module.ts
  users.service.ts
  entities/
    user.entity.ts
```

구현 흐름:

1. `AuthController.login()`이 `LoginDto`를 받는다.
2. `LoginService.login()`이 `UsersService.findByEmail()`을 호출한다.
3. user가 없으면 `INVALID_CREDENTIALS`를 반환한다.
4. 비밀번호를 비교한다.
5. 비밀번호가 틀리면 `INVALID_CREDENTIALS`를 반환한다.
6. `AuthService`를 통해 accessToken과 refreshToken을 발급한다.
7. 로그인 응답을 반환한다.

이 구조를 따르면 로그인 구현 이후 signup, refresh, logout을 추가해도 auth 폴더가 지저분해지지 않는다.
