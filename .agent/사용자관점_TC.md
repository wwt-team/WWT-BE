# 사용자관점 테스트 기준

## 목적

이 문서는 기능을 API 단위로 잘게 나누기보다, 사용자가 실제로 서비스를 사용하는 흐름 기준으로 테스트 시나리오를 정리한다.

핵심 기준은 아래 4가지다.

1. 사용자가 실제로 어떤 순서로 기능을 밟는가
2. 실패했을 때 다음 행동을 이해할 수 있는가
3. 권한과 자산이 안전하게 보호되는가
4. 한 번의 행동 결과가 다음 화면과 상태에 일관되게 반영되는가

## 면접에서 설명할 기준

테스트 케이스는 API 개수 기준으로 먼저 자르지 않는다.  
사용자가 처음 진입해서 인증하고, 자기 정보를 확인하고, 상품을 다루고, 거래 요청을 만들고, 채팅으로 이어지는 실제 흐름을 먼저 본다.

그래서 테스트는 아래 질문에 답하는 형태로 작성한다.

- 사용자는 어디에서 가장 자주 막히는가
- 실패 메시지는 다음 행동을 안내하는가
- 권한 없는 접근은 제대로 차단되는가
- 상태 전이는 다음 단계에서 일관되게 보이는가

## 전체 도메인 순서

문서와 사용자 흐름 기준 순서는 아래로 고정한다.

1. Auth
2. Users
3. Products
4. Trade Requests
5. Chats
6. Deploy QA

## Fast와 Complex를 나누는 기준

### Fast

Fast는 사용자가 가장 자주 밟는 핵심 흐름이다.

예:

- 가입하고 로그인한다
- 내 정보를 조회한다
- 상품을 등록하고 수정한다
- 거래 요청을 생성한다
- 채팅방을 만들고 메시지를 보낸다

즉 Fast는 “이 서비스가 기본적으로 되는가”를 확인하는 축이다.

### Complex

Complex는 사용자가 평범하게 사용한다고 생각하지만, 실제로는 시스템 규칙이 꼬이기 쉬운 지점을 검증하는 축이다.

예:

- 같은 요청을 여러 번 시도한다
- 이미 끝난 상태를 다시 건드린다
- 한 번의 행동이 여러 상태를 함께 바꾼다
- 실시간 이벤트가 참여자에게만 전달되어야 한다
- 토큰 재사용이나 세션 연장 규칙이 숨어 있다

즉 Complex는 “기능이 된다”보다 “복잡한 규칙이 끝까지 일관되게 유지되는가”를 확인하는 축이다.

## 1. Auth

### 사용자 흐름

1. `signup/email-code`
2. `signup/email-code/verify`
3. `signup`
4. `login`
5. `accessToken`으로 보호 API 접근 확인
6. `refresh`
7. `logout`

### 사용자 관점 포인트

- 이메일 인증은 실제 가입 단계로 자연스럽게 이어져야 한다.
- 로그인 후 받은 access token은 바로 보호 API에서 동작해야 한다.
- refresh는 세션 연장 흐름으로 동작해야 한다.
- logout 이후 기존 refresh token은 더 이상 동작하면 안 된다.

### Complex를 잡는 사용자 상황

- 로그인 상태를 오래 유지하려고 refresh를 여러 번 시도한다.
- 로그아웃 후에도 앱이 예전 refresh token을 다시 보낼 수 있다.
- 오래된 refresh token이 남아 있어 재사용이 발생할 수 있다.

## 2. Users

### 사용자 흐름

1. `login`
2. `GET /api/users/me`
3. `GET /api/users/me/products`

### 사용자 관점 포인트

- 로그인한 사용자는 먼저 자기 정보를 확인할 수 있어야 한다.
- 내 상품 목록은 내 계정 컨텍스트와 연결되어 자연스럽게 이어져야 한다.
- 인증이 없으면 내 정보나 내 상품 목록은 조회되면 안 된다.

### Complex를 잡는 사용자 상황

- 내 상품이 많아져 페이지를 넘기며 확인한다.
- 페이지가 바뀌어도 중복 없이 정확히 보이길 기대한다.

## 3. Products

### 판매자 흐름

1. `login`
2. 상품 등록
3. 상품 목록/상세 확인
4. 상품 수정
5. 상품 상태 변경
6. 상품 삭제

### 구매자 흐름

1. `login`
2. 상품 목록 조회
3. 상품 검색
4. 상품 상세 조회

### 사용자 관점 포인트

- 상품 등록 후 목록과 상세에서 같은 정보가 보여야 한다.
- 잘못된 제목, 가격, 이미지 URL은 사용자가 이해 가능한 방식으로 막혀야 한다.
- 판매자만 수정/삭제/상태 변경을 할 수 있어야 한다.

### Complex를 잡는 사용자 상황

- 상품 상태를 바꿨는데 목록과 상세가 서로 다르게 보인다.
- 검색 조건을 여러 개 걸었는데 결과가 예상과 다르다.
- 같은 판매자가 같은 제목과 같은 설명의 상품을 실수로 두 번 등록하려고 할 수 있다.

## 4. Trade Requests

### 구매자 흐름

1. `login`
2. 상품 상세 조회
3. 거래 요청 생성
4. 내 거래 요청 목록 조회

### 판매자 흐름

1. `login`
2. 상품별 거래 요청 목록 조회
3. 거래 요청 수락 또는 거절

### 사용자 관점 포인트

- 자기 상품에는 거래 요청을 보내면 안 된다.
- 이미 판매 완료된 상품에는 요청하면 안 된다.
- 중복 대기 요청은 차단되어야 한다.
- 수락 후 상품 상태는 `RESERVED`로 일관되게 반영되어야 한다.

### Complex를 잡는 사용자 상황

- 구매자가 버튼을 여러 번 눌러 같은 요청을 반복한다.
- 판매자가 이미 처리한 요청을 다시 누른다.
- 요청을 수락했는데 상품 상태가 바뀌지 않으면 혼란이 생긴다.

## 5. Chats

### REST 흐름

1. `login`
2. 채팅방 생성
3. 채팅방 목록 조회
4. 메시지 목록 조회
5. 메시지 전송
6. 메시지 수정
7. 메시지 삭제

### 실시간 흐름

1. WebSocket 연결
2. 채팅방 입장
3. 메시지 생성 이벤트 수신
4. 메시지 수정 이벤트 수신
5. 메시지 삭제 이벤트 수신

### 사용자 관점 포인트

- 채팅방은 상품과 상대방 기준으로 일관되게 생성되어야 한다.
- 참여자만 메시지를 조회/전송/수정/삭제할 수 있어야 한다.
- 삭제된 메시지는 다시 수정되면 안 된다.
- 실시간 이벤트는 참여자에게만 전달되어야 한다.

### Complex를 잡는 사용자 상황

- 같은 상품, 같은 상대와 채팅방을 여러 번 만들려고 한다.
- 삭제한 메시지를 다시 수정하려고 한다.
- 참여자가 아닌 방에 접속하려고 한다.
- 상대방에게 실시간으로 보이지 않거나, 반대로 불필요한 사람에게 이벤트가 전파될 수 있다.

## 6. Deploy QA

### 배포 환경 흐름

1. 서버 접근 확인
2. 로그인
3. Users 흐름 확인
4. Products 흐름 확인
5. Trade Requests 흐름 확인
6. Chats 흐름 확인
7. 운영 환경 요소 확인

### 사용자 관점 포인트

- EC2, Nginx, PM2, DB 연결이 실제 배포 환경에서 안정적으로 동작해야 한다.
- CORS, SMTP, WebSocket 같은 운영 의존 요소가 실제 환경에서 정상 동작해야 한다.
- 브라우저에서 외부 주소로 접근했을 때 주요 기능 흐름이 모두 이어져야 한다.

### Complex를 잡는 사용자 상황

- 서버가 재부팅돼도 서비스가 계속 살아 있어야 한다.
- 브라우저에서 CORS 없이 API를 호출할 수 있어야 한다.
- 실제 메일이 와야 하고, 실제 소켓도 붙어야 한다.

## 테스트 케이스 작성 원칙

각 기능은 최소 아래 5축으로 검증한다.

1. 정상 성공
2. 입력 실패
3. 인증 실패
4. 권한 실패
5. 상태 전이/일관성

예: 상품 수정

- 정상: 판매자가 상품을 수정한다.
- 입력 실패: 가격이 잘못되면 실패한다.
- 인증 실패: 로그인하지 않으면 실패한다.
- 권한 실패: 다른 사용자는 수정할 수 없다.
- 일관성: 수정 후 상세 조회에서 변경 내용이 보인다.
